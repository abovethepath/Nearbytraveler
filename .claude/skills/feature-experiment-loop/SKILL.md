# Feature to Experiment Loop

Convert product ideas into learning loops.

## Goal
Prevent Nearby Traveler from building features without testing whether they solve a real user problem or improve traction.

## Process
For any proposed feature, produce:

1. **Feature idea**
2. **User problem**
3. **Why this matters now**
4. **Riskiest assumption**
5. **Fastest MVP test**
6. **Success metric**
7. **Failure signal**
8. **What to learn**
9. **Next decision if it works**
10. **Next decision if it fails**

## Nearby Traveler strategic lens
Prefer features that improve one of these:
- connection speed
- trust
- responsiveness
- local density
- reasons to return
- immediate activity

Deprioritize features that:
- look impressive but don't create activity
- only matter after scale
- increase complexity without improving connection

## Output template
### Feature
### User problem
### Risky assumption
### MVP test
### Metric
### What success means
### What failure means
### Recommendation

## Examples of strong test framing
- "Will seeing who arrives in the next 3 days increase message starts?"
- "Will a Tonight in This City module increase same-day engagement?"
- "Will a live city feed make empty-network cities feel active enough to retain users?"
